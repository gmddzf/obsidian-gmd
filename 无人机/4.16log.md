	# 🚁 无人机编程平台开发与排错日志
 
## 一、 业务逻辑层：状态误判导致程序“假死”
 
### 🐛 问题1：单条指令失败导致后续全部瘫痪
- **现象**：无人机执行某个动作（如起飞没放平）失败后，后面拖入的所有积木运行时都提示“❌ 请先连接无人机”，但明明WiFi还连着。
- **根本原因**：在 `run_drone_command` 函数中，一旦某个指令抛出异常，代码里写了 `is_connected = False`。程序把“指令执行超时/失败”误判成了“物理断开连接”，导致全局状态锁死。
- **改进方案**：
  - **剥离状态**：指令执行失败（如超时）仅仅代表这次动作没成功，不代表网络断开。移除异常处理中的 `is_connected = False`。
  - **状态重置**：只有在主动调用 `connect_drone()` 失败时，才将 `is_connected` 设为 False；只有在收到物理断开事件时才改变连接状态。

## 二、 部署与打包层：从“源码”到“独立exe”的坑
 
### 🐛 问题2：用户双击 `.py` 文件报错“无法运行”
- **现象**：把源码发给同事，同事双击 `app.py`，Windows 报错“此应用无法在你的电脑上运行”。
- **根本原因**：同事电脑没有配置 Python 环境变量，或者 Windows 试图用错误的解释器去打开它。
- **改进方案**：坚决不要让非技术人员运行源码。必须使用 `.bat` 脚本启动，或者最终打包成 `.exe`。
 
### 🐛 问题3：打包成 exe 后，照片保存到了系统缓存目录找不到
- **现象**：打包后运行，点击拍照提示成功，但在 exe 同级目录下找不到 `photos` 文件夹和照片。
- **根本原因**：Python 打包成单文件 exe 运行时，会把代码解压到系统的临时目录（`AppData\Local\Temp`），原代码中使用的 `os.path.abspath(__file__)` 获取的是临时目录的路径，导致文件夹建到了缓存里。
- **改进方案**：动态判断运行环境，优先使用 exe 所在的目录。
  ```python
  if getattr(sys, 'frozen', False):
      BASE_DIR = os.path.dirname(sys.executable) # 打包后：取exe所在路径
  else:
      BASE_DIR = os.path.dirname(os.path.abspath(__file__)) # 源码运行：取脚本路径
	### 🐛 问题4：打包成 exe 后双击直接闪退，毫无提示
	- **现象**：运行 exe 后黑框一闪而过，无法看到任何报错信息。
	- **根本原因**：PyInstaller 默认不会自动打包所有隐式依赖（如 OpenCV 的底层库），导致程序启动时依赖缺失直接崩溃。且 `if __name__ == '__main__':` 没有拦截错误，进程瞬间结束。
	- **改进方案**：
	  1. **防闪退拦截**：在主入口加 `try...except...finally`，并在 `finally` 中加 `input("按回车键退出...")`，强行使窗口停留。
	  2. **强制包含依赖**：打包命令增加隐式引用参数 `--hidden-import=cv2 --hidden-import=numpy` 等。
	  3. **代码示例**：
	      if __name__ == '__main__':
	          try:
	              app.run(debug=False, host='0.0.0.0', port=8080)
	          except Exception as e:
	              print(f"❌ 程序运行出错: {e}")
	          finally:
	              input("程序已停止运行，按回车键退出...")

	## 三、 网络通信层：无人机连接的“玄学”所在
	### 🐛 问题5：换电脑后连上无人机WiFi，代码报连接失败，且控制台打印了新IP
	- **现象**：新电脑连着 Tello WiFi，运行提示连接失败，黑框还多显示了一个新IP。
	- **根本原因**：
	  1. 关于IP：代码里打印的是电脑的局域网IP，供手机浏览器访问用的，换电脑必然变，与无人机连接无关。
	  2. 关于失败：电脑连上 WiFi 后，网卡需要时间获取 IP 地址（DHCP分配）。如果程序启动太快，电脑还没拿到 `192.168.10.x` 的 IP，发出的指令无人机收不到。
	- **改进方案**：
	  - 在发送连接指令前，先加一个**网络就绪检测**（自动 Ping `192.168.10.1`）。
	  - 增加无人机库的超时等待时间：`tello.CONNECT_TIMEOUT = 10`，给网卡响应留足余量。
	### 🐛 问题6：Ping 得通 192.168.10.1，但程序还是连不上
	- **现象**：手动 Ping 无人机 IP 是通的，说明网络没断，但代码依然连接失败。
	- **根本原因**：混淆了 ICMP 协议与 UDP 协议。Ping 通了只代表“路修好了（网络层连通）”，但无人机通信使用的是 **UDP 协议（端口 8889）**，UDP 报文被拦截了。
	- **排查方向（三大元凶）**：
	  1. **防火墙区别对待**：Windows 把无密码的 Tello WiFi 识别为“公用网络”，放行了 Ping（ICMP），但拦截了程序使用的 UDP 端口。**解法**：关闭防火墙测试，或在防火墙放行规则中勾选“专用+公用”。
	  2. **虚拟网卡抢路**：电脑装的 VPN、游戏加速器、VMware 虚拟网卡，可能会把发往 `192.168.10.1` 的 UDP 数据包路由到虚拟网卡上。**解法**：在“网络连接”中禁用除 Wi-Fi 以外的所有虚拟网卡。
	  3. **端口残留占用**：上一次程序异常退出，导致电脑本地的 8889 端口未释放。**解法**：任务管理器杀掉残留进程。

	## 💡 未来排错速查口诀（建议收藏）
	以后遇到无人机连不上，按这个顺序查：
	1. **看现象**：程序报错是 `timed out`（网络不通/防火墙拦截）、`address already in use`（端口占用）、还是 `network is unreachable`（根本没连WiFi）？
	2. **查WiFi**：电脑是不是**只**连了 Tello 的 WiFi？有没有同时连着有线网？
	3. **等几秒**：连上 WiFi 后，心里默数 5 秒再点运行，给网卡拿 IP 的时间。
	4. **关防火**：最直接的验证方法，连上 Tello WiFi 后直接关掉电脑防火墙测试，如果通了，就是防火墙规则问题。
	5. **断虚拟**：如果公司电脑怎么都连不上，去网络设置里把 VMware 和 VPN 网卡全禁用。
